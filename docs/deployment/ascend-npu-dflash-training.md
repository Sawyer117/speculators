# Training a Qwen3-8B DFlash Speculator on Ascend NPU

End-to-end guide for training a **DFlash** draft model for **Qwen3-8B** on
**Huawei Ascend NPU**, using speculators' **online training** (vLLM serves the
target and extracts hidden states on the fly). Hyperparameters are aligned to the
SpecForge `docs/ascend_npu/run_qwen3_8b_dflash_npu.sh` reference run.

> **Status (June 2026).** Prerequisite install is verified (see
> [`ascend-npu-conda.md`](./ascend-npu-conda.md)). The hyperparameter alignment and
> data-format findings below are confirmed from source. The **run commands are
> being validated on NPU** — the open risk is whether vllm-ascend's NPU model
> runner supports the `extract_hidden_states` path (see §5). Sections marked
> ⏳ are pending NPU confirmation.

## 1. Prerequisites

- Install chain from [`ascend-npu-conda.md`](./ascend-npu-conda.md) working:
  vLLM 0.20.2 (empty build) + vllm-ascend 0.20.2rc1 + torch/torch-npu 2.10.0 +
  triton-ascend 3.2.1, and an editable speculators.

  `/share/canada_group_folder/dataset/perfectblend_train_regen.jsonl`.

## 2. Hyperparameter alignment

Source of truth: SpecForge `docs/ascend_npu/run_qwen3_8b_dflash_npu.sh` (HF backend)
and the draft config it references, `configs/qwen3-8b-dflash.json`.

| Concept | ascend_npu reference | speculators flag | aligned value |
|---|---|---|---|
| epochs | `NUM_EPOCHS=6` | `--epochs` | `6` |
| learning rate | `LR=6e-4` | `--lr` | `6e-4` |
| sequence length | `MAX_LENGTH=3072` | `--total-seq-len` + prepare `--seq-length` | `3072` ¹ |
| anchors | `NUM_ANCHORS=512` | `--max-anchors` | `512` |
| block size | `BLOCK_SIZE=16` | `--block-size` | `16` |
| grad clip | `MAX_GRAD_NORM=1.0` | (hardcoded 1.0 in trainer) | matches ✅ |
| draft layers | `num_hidden_layers=5` | `--num-layers` | `5` |
| draft backbone | `model_type=qwen3` | `--draft-arch` | `qwen3` ² |
| target layers | `[1,9,17,25,33]` | `--target-layer-ids` (and launch_vllm) | `1 9 17 25 33` |
| draft vocab | full `151936` | omit `--draft-vocab-size` | full vocab |
| mask token | `mask_token_id=151669` | `--mask-token-id` | `151669` |
| activation | `hidden_act=silu` | `--draft-hidden-act` | `silu` |

¹ Semantics differ: SpecForge pads one sample per batch to `MAX_LENGTH`;
speculators packs multiple samples up to a `--total-seq-len` token budget
(MultipackDistributedBatchSampler). Same number, different batching.

² speculators' `train.py` warns that a `qwen3` draft is "not yet supported in
vLLM for inference". Training is unaffected; revisit before deployment, or switch
to `--draft-arch llama` if vLLM-deploy compatibility matters more than matching
the reference config.

### 2.1 Gaps (no speculators CLI equivalent — using defaults)

| ascend_npu | value | speculators behavior |
|---|---|---|
| `LOSS_DECAY_GAMMA` | 7.0 | mechanism exists (`dflash_loss_decay`) but gamma is **not** exposed via CLI; defaults to **4.0**. Exposing it requires patching `train.py` + DFlash `get_trainer_kwargs`. |
| `WARMUP_RATIO` | 0.04 | speculators uses absolute `--scheduler-warmup-steps`, not a ratio; default used. |
| `BATCH_SIZE` / `ACCUMULATION_STEPS` | 1 / 1 | no batch/accum flags; token-budget packing via `--total-seq-len`. |

## 3. NPU runtime environment

Export these in every shell that runs vLLM or training (from the ascend_npu
scripts + the OpenMP guards that prevent the `ParallelOpenMP.cpp:64` crash):

```bash
export OMP_PROC_BIND=false OMP_NUM_THREADS=1 MKL_NUM_THREADS=1 VE_OMP_NUM_THREADS=1
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True
export TASK_QUEUE_ENABLE=2 ACLNN_CACHE_LIMIT=100000 NPU_ASD_ENABLE=0 ASCEND_LAUNCH_BLOCKING=0
```

> ⚠️ **`TASK_QUEUE_ENABLE` per process (verified gotcha).** The value `2` (from the
> SpecForge ascend_npu scripts) is **incompatible with vLLM's NPU graph capture** —
> a graph-mode vLLM serve dies with
> `Do not support TASK_QUEUE_ENABLE = 2 during NPU graph capture, please export
> TASK_QUEUE_ENABLE=1/0`. So:
> - **vLLM serve (§5):** use `TASK_QUEUE_ENABLE=1` (or pass `--enforce-eager`, which
>   skips graph capture and sidesteps the conflict entirely).
> - **Training (§6):** keep `TASK_QUEUE_ENABLE=2` — plain eager FSDP does no graph
>   capture, and `2` is faster.

### 3.1 Device split (this is the big difference from SpecForge)

SpecForge runs single-process: the target lives **in** the training process (HF
replica per NPU, or sglang), so all 8 NPUs do "target forward + draft training"
together. speculators online runs **two processes** — vLLM serves the target, the
trainer runs separately — so the 8 NPUs must be **split** between them. You cannot
put 8-way draft FSDP and target serving on the same cards.

Default split (uses all 8): **vLLM target on `0,1` (TP=2) + draft FSDP on `2-7`
(6 cards)** — 2 serve, 6 train.

**This split is highly tunable** — it's a throughput tradeoff between target-side
hidden-state extraction and draft-side data parallelism:

| Lever | Effect |
|---|---|
| vLLM `--tensor-parallel-size` (1/2/4) | An 8B target fits TP=1 → frees a card for training (7-way FSDP). Larger targets need more TP. |
| vLLM `--data-parallel-size N` | Multiple target replicas → more concurrent hidden-state requests, if extraction is the bottleneck. |
| training `--nproc_per_node` | More cards = more FSDP data parallelism = faster training, if compute is the bottleneck. |

Rule of thumb: if the trainer is starved waiting on hidden states, give vLLM more
cards (TP or DP); if vLLM sits idle, give the trainer more cards. Measure which
side is idle, then rebalance.

> To run **true 8-way draft training** (exactly matching SpecForge's 8 cards), use
> the **offline** path (§7): generate hidden states with all 8 NPUs first, stop
> vLLM, then train the draft on all 8. Online can't — it always reserves cards for
> serving.

## 4. Data: regen JSONL → tokenized dataset

The regenerate step (SpecForge `regenerate_train_data.py`) already produced the
target-aligned text. speculators' `prepare_data.py` is a **different** step — it
does NOT regenerate; it tokenizes, builds loss masks, and writes an Arrow dataset
that the trainer reads. It is mandatory: `train.py` reads an `ArrowDataset`, not
raw JSONL.

**Format compatibility (confirmed):** speculators' custom-JSONL path applies no
normalization, so the file must already use the `conversations` top-level key with
per-turn `role`/`content` (or sharegpt `from`/`value`). The verified data has
exactly `{"id", "conversations":[{"role","content"}...], "status"}` — directly
compatible, no conversion needed. Thinking is inline in `content` as
`<think>...</think>`, so it's trained as part of the assistant tokens (correct for
a thinking-enabled target). `--seq-length 3072` truncates long reasoning samples
(aligned to the reference `MAX_LENGTH=3072`).

### Step 1 — prepare data (CPU only)

```bash
python scripts/prepare_data.py \
  --model /share/canada_group_folder/ckpt/Qwen3-8B \
  --data /share/canada_group_folder/dataset/perfectblend_train_regen.jsonl \
  --output ./output/qwen3-8b-dflash-npu \
  --seq-length 3072 \
  --max-samples 200 \
  --trust-remote-code
```

Confirm: logs show `Loaded N samples` (N > 0), and the output dir contains
`*.arrow`, `dataset_info.json`, `state.json`, `token_freq.pt`. If you see
`No conversations key found` or 0 processed samples, the Qwen3 thinking assistant
boundary wasn't matched — pass an explicit `--assistant-pattern`. Drop
`--max-samples 200` for the full run once the pipeline is verified.

## 5. Step 2 — launch vLLM (target serving + hidden-state extraction) ✅ startup verified

The online path relies on two vLLM-core features (both present in the local
v0.20.2 build: `extract_hidden_states` + `ExampleHiddenStatesConnector`). The
connector is device-agnostic; the extraction hooks live partly in the model runner.
**Verified on NPU:** with `--enforce-eager`, vllm-ascend accepts the
`extract_hidden_states` speculative_config + connector and reaches
`Application startup complete`. (Hidden-state files actually appearing during
training is confirmed in §6.)

```bash
# terminal 1, in the speculators repo root (NOT next to a vllm/ source dir)
export OMP_PROC_BIND=false OMP_NUM_THREADS=1 MKL_NUM_THREADS=1 VE_OMP_NUM_THREADS=1
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True
export TASK_QUEUE_ENABLE=1 ACLNN_CACHE_LIMIT=100000 NPU_ASD_ENABLE=0 ASCEND_LAUNCH_BLOCKING=0

ASCEND_RT_VISIBLE_DEVICES=0,1 python scripts/launch_vllm.py \
  /share/canada_group_folder/ckpt/Qwen3-8B \
  --target-layer-ids 1 9 17 25 33 --no-include-last-layer \
  --hidden-states-path ./tmp/hs_qwen3_dflash \
  -- --tensor-parallel-size 2 --port 8000 --enforce-eager
```

Wait for `Application startup complete`.

- `--enforce-eager` skips NPU graph capture — the verified path. Dropping it for
  graph-mode speedup is a tuning step (then `TASK_QUEUE_ENABLE=1` is required, see
  §3 note); graph mode + `extract_hidden_states` on NPU is not yet validated.
- If startup ever fails citing `extract_hidden_states` / spec_decode / model_runner,
  the NPU runner doesn't support this path → fall back to **offline** (§7).

> ⚠️ **Write path = read path.** This `--hidden-states-path` (where the connector
> **writes**) must equal the trainer's `--hidden-states-path` (where it **reads**,
> §6), and both must resolve to the same absolute dir. With a relative `./tmp/...`
> path, **launch both terminals from the same cwd**, or export an absolute path once
> (`export HS_DIR="$(pwd)/tmp/hs_qwen3_dflash"`) and use `$HS_DIR` in both. A
> mismatch silently stalls online training (vLLM writes, trainer reads elsewhere).

## 6. ⏳ Step 3 — train the DFlash draft (FSDP)

```bash
# terminal 2, 6 training cards (2-7); vLLM holds 0,1. TASK_QUEUE_ENABLE=2 ok here
# (eager FSDP, no graph capture). --hidden-states-path MUST match §5 (same cwd).
export OMP_PROC_BIND=false OMP_NUM_THREADS=1 MKL_NUM_THREADS=1 VE_OMP_NUM_THREADS=1
export PYTORCH_NPU_ALLOC_CONF=expandable_segments:True
export TASK_QUEUE_ENABLE=2 ACLNN_CACHE_LIMIT=100000 NPU_ASD_ENABLE=0 ASCEND_LAUNCH_BLOCKING=0

ASCEND_RT_VISIBLE_DEVICES=2,3,4,5,6,7 torchrun \
  --nproc_per_node 6 --nnodes 1 --node_rank 0 \
  --master_addr 127.0.0.1 --master_port 29533 \
  scripts/train.py \
  --verifier-name-or-path /share/canada_group_folder/ckpt/Qwen3-8B \
  --data-path ./output/qwen3-8b-dflash-npu \
  --vllm-endpoint http://localhost:8000/v1 \
  --hidden-states-path ./tmp/hs_qwen3_dflash \
  --save-path ./output/qwen3-8b-dflash-npu/checkpoints \
  --speculator-type dflash \
  --draft-arch qwen3 \
  --num-layers 5 \
  --block-size 16 \
  --max-anchors 512 \
  --target-layer-ids 1 9 17 25 33 \
  --mask-token-id 151669 \
  --draft-hidden-act silu \
  --epochs 6 \
  --lr 6e-4 \
  --total-seq-len 3072 \
  --use-off-policy-tokens \
  --logger tensorboard \
  --on-missing generate \
  --on-generate delete \
  --trust-remote-code
```

- `--use-off-policy-tokens`: required for regenerated data (per `train.py` help).
  Drop it only if the data is not target-regenerated.
- `--master_addr 127.0.0.1` (not `--standalone`): avoids torchrun rendezvous
  resolving a non-routable external hostname (same fix as the ascend_npu script).
- speculators' distributed init is device-agnostic (`torch.accelerator` +
  `get_default_backend_for_device`), so it auto-selects HCCL on NPU. If it can't
  find the NPU, add `import torch_npu` at the top of `train.py`.
